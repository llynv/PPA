Analysis Engine (analysis.ts) — what's broken:
1. Hardcoded frequency buckets — The "optimal" action is just 3 hardcoded tiers: strength > 0.7 → raise, 0.35-0.7 → call, < 0.35 → fold. Real GTO depends on position, board texture, bet sizing, stack depth, number of players, and whether you're IP or OOP.

5. Random noise in "optimal" frequencies — The ±5% noise means the same exact situation produces different "optimal" advice each time. GTO frequencies should be deterministic.

Design Section 2: Monte Carlo Equity Calculator (equity.ts)
What it does: Given hero's hole cards, the community cards, and the number of opponents, simulate N random rollouts to estimate hero's probability of winning.
Algorithm:
1. Remove known cards (hero's hole + community) from the deck
2. For each simulation (N=1000 default):
   - Deal random hole cards to each opponent from remaining cards
   - Deal remaining community cards to complete the board
   - Evaluate all hands, determine winner
   - Tally wins/ties for hero
3. equity = (wins + ties/numPlayers) / N
Performance: With the existing evaluateHand() (C(7,5)=21 combinations per hand), 1000 sims with 4 opponents = ~84,000 hand evaluations. On modern JS engines, each evaluation takes ~1-5 microseconds, so total 80-400ms. For AI decisions during play, we can reduce to 500 sims (40-200ms) which is fast enough to feel instant.
API:
interface EquityResult {
  equity: number;       // 0-1, probability of winning
  samples: number;      // number of simulations run
  wins: number;
  ties: number;
  losses: number;
}
function calculateEquity(
  holeCards: Card[],
  communityCards: Card[],
  numOpponents: number,
  samples?: number,     // default 1000
): EquityResult
Key detail — equity vs. hand strength: The current system's strength (0-1 normalized hand rank) measures absolute hand quality. Equity measures probability of winning against opponents. A pair of aces on a board of 4h 5h 6h 7h has high absolute strength (pair) but low equity (any heart or 8 or 3 beats it). Equity is what matters for decisions.

Design Section 4: Draw Detection & Board Texture (draws.ts, board.ts)
Draw detection — Analyze hero's hole cards + community cards:
- Flush draw: 4 cards of same suit (9 outs, ~35% equity on flop)
- Open-ended straight draw (OESD): 4 consecutive ranks with room on both ends (8 outs, ~31%)
- Gutshot: Inside straight draw (4 outs, ~17%)
- Combo draw: Flush draw + straight draw combined (12-15 outs, ~45-54%)
- Backdoor draws: 3 to a flush or 3 to a straight (partial equity)
interface DrawInfo {
  flushDraw: boolean;
  flushDrawOuts: number;
  oesD: boolean;           // open-ended straight draw
  gutshot: boolean;
  straightDrawOuts: number;
  backdoorFlush: boolean;
  backdoorStraight: boolean;
  totalOuts: number;
  drawEquity: number;      // estimated equity from draws alone
}
Board texture analysis — Categorize the board:
- Wetness: How many draws are possible (monotone/two-tone/rainbow, connected/disconnected)
- Pairing: Board has a pair (full house possible)
- High card texture: Broadway-heavy vs low cards
- Coordinated vs dry: Jh Th 9s (very wet) vs Kd 7c 2h (very dry)
interface BoardTexture {
  wetness: 'dry' | 'semi-wet' | 'wet' | 'very-wet';
  isMonotone: boolean;
  isTwoTone: boolean;
  isRainbow: boolean;
  isPaired: boolean;
  isTrips: boolean;
  highCardCount: number;    // number of broadway cards (T+)
  connectedness: number;    // 0-1, how connected the board is
  possibleStraights: number;
  possibleFlushes: boolean;
}
Board texture matters for bet sizing — on wet boards, you should bet larger to deny equity to draws. On dry boards, smaller bets work.

Design Section 5: Decision Framework (decision.ts)
This is the core — a single function that evaluates any poker situation and returns the optimal action with frequencies and EV.
Inputs:
interface DecisionContext {
  holeCards: Card[];
  communityCards: Card[];
  position: Position;
  round: BettingRound;
  pot: number;
  toCall: number;
  currentBet: number;      // highest bet on the table
  stack: number;            // hero's remaining stack
  bigBlind: number;
  numActivePlayers: number;
  numPlayersInHand: number;
  isFirstToAct: boolean;
  facingRaise: boolean;     // did someone raise before us
  raiserPosition?: Position;
  actionHistory: PlayerAction[];  // all actions this hand
}
Output:
interface DecisionResult {
  optimalAction: ActionType;
  optimalAmount?: number;
  frequencies: { fold: number; call: number; raise: number };
  reasoning: string;        // human-readable explanation
  equity: number;           // Monte Carlo equity
  potOdds: number;
  impliedOdds: number;
  spr: number;              // stack-to-pot ratio
  draws: DrawInfo;
  boardTexture: BoardTexture;
  evByAction: {
    fold: number;
    call: number;
    raise: number;
  };
}
Decision logic (preflop):
1. Determine position → look up range charts
2. If no action before us → check if hand is in open-raise range
3. If facing a raise → check 3-bet range, call range
4. If facing a 3-bet → check 4-bet range, call range
5. Frequencies are based on range inclusion + position
Decision logic (postflop):
1. Calculate equity via Monte Carlo
2. Detect draws
3. Analyze board texture
4. Calculate pot odds: toCall / (pot + toCall)
5. Calculate SPR: stack / pot
6. EV of each action:
   - EV(fold) = 0 (always)
   - EV(call) = equity * (pot + toCall) - (1 - equity) * toCall
   - EV(raise) = more complex: fold equity from opponents + equity when called * larger pot - cost of raise
7. Bet sizing:
   - Dry board → 25-33% pot
   - Semi-wet board → 50-66% pot
   - Wet board → 75-100% pot
   - Strong draws → overbet (125%+ pot)
8. Frequencies derived from EV comparison:
   - If EV(raise) >> EV(call) → high raise frequency
   - If EV(call) > 0 but EV(raise) ≈ EV(call) → mixed call/raise
   - If all EVs ≤ 0 → high fold frequency
The reasoning field is key for the analysis UI — it explains why a particular action is optimal: "You have 45% equity with a combo draw. Pot odds require 28%. Calling is +EV at $4.30/hand. Raising is even better because you have fold equity against one-pair hands."

Personality	Deviation from optimal
TAG	Plays ~95% of optimal ranges, slightly tighter. Rarely bluffs.
LAG	Opens ~130% of optimal ranges, 3-bets wider. Higher bluff frequency.
Tight-passive	Opens ~80% of optimal ranges. Rarely raises, mostly calls. Checks strong hands.

Added dimension: tilt/adaptation — Each AI tracks win/loss over recent hands. After losing big, LAG players might loosen up more (tilting). TAG players might tighten up. This creates dynamic gameplay rather than static opponents.

# AI Player & Analysis Engine Overhaul — Design
**Date:** 2026-03-19
**Status:** Approved
## Problem
Both the AI player (`ai.ts`) and analysis engine (`analysis.ts`) use oversimplified heuristics:
- **No equity calculation** — Decisions use `getBestHand().strength` (absolute hand rank) instead of actual probability of winning. A flush draw with 45% equity shows as "High Card" strength ~0.2.
- **No position awareness** — AI plays the same from UTG as from the button. Analysis doesn't consider position.
- **No draw detection** — Neither system recognizes flush draws, straight draws, or combo draws.
- **Hardcoded frequency buckets** — Analysis uses 3 static tiers (strong/medium/weak) with random noise. Same spot produces different "optimal" advice each run.
- **No bet sizing analysis** — Only checks fold/call/raise category, ignoring sizing.
- **No opponent modeling** — AI ignores stack-to-pot ratio, board texture, and prior action.
- **Invented EV formula** — `evLoss = freqDiff * potInBB * (1 - strength) * 0.5` has no theoretical basis.
## Architecture
Build a shared poker engine (`src/lib/poker-engine/`) consumed by both AI and analysis:
```
src/lib/poker-engine/
├── equity.ts        # Monte Carlo equity calculator
├── ranges.ts        # Preflop hand range definitions by position
├── position.ts      # Position utilities (UTG through BB)
├── draws.ts         # Draw detection (flush, OESD, gutshot, combo, backdoor)
├── board.ts         # Board texture analysis (wet/dry, paired, monotone)
├── decision.ts      # Core decision framework (EV-based)
└── index.ts         # Re-exports
```
- **AI** calls the decision framework with personality-based deviations
- **Analysis** calls the same framework without deviations to compute optimal play
- Both share identical equity, range, draw, and board logic

```typescript
    fold: number;
    call: number;
    raise: number;
  };
}
function evaluateDecision(ctx: DecisionContext): DecisionResult
```
### Preflop logic:
1. Determine position → look up range charts
2. No prior action → check open-raise range
3. Facing raise → check 3-bet/call ranges
4. Facing 3-bet → check 4-bet/call ranges
### Postflop logic:
1. Calculate equity via Monte Carlo
2. Detect draws, analyze board texture
3. Calculate pot odds: `toCall / (pot + toCall)`
4. Calculate SPR: `stack / pot`
5. EV of each action:
   - `EV(fold) = 0`
   - `EV(call) = equity × (pot + toCall) - (1 - equity) × toCall`
   - `EV(raise) = foldEquity × pot + (1 - foldEquity) × [equity × (pot + raiseSize) - (1 - equity) × raiseSize]`
6. Bet sizing by board texture:
   - Dry → 25-33% pot
   - Semi-wet → 50-66% pot
   - Wet → 75-100% pot
7. Frequencies derived from EV comparison
### Reasoning generation:
Every decision includes a human-readable explanation: "You have 45% equity with a combo draw. Pot odds require 28%. Calling is +EV at $4.30. Raising is better because you have fold equity against one-pair hands."
## AI Player Overhaul (`ai.ts`)
Thin wrapper over decision engine with personality deviations:
```typescript
function getAIDecision(params: AIDecisionParams): AIDecisionResult {
  const optimal = evaluateDecision(buildContext(params));
  return applyPersonalityDeviations(optimal, params.player.personality);
}
```
### Personality deviations:
| Personality | Range adjustment | Aggression | Bluff freq | Behavior |
|-------------|-----------------|------------|------------|----------|
| TAG | 95% of optimal | Slightly above optimal | Low (5%) | Tight, aggressive when entering |
| LAG | 130% of optimal | High | High (20%) | Wide opens, frequent 3-bets |
| Tight-passive | 80% of optimal | Very low | Minimal (3%) | Calls too much, rarely raises |
| Loose-passive | 150% of optimal | Low | Low (8%) | Calls everything, never folds |
### Dynamic adaptation:
Each AI tracks recent results. After losing big, LAG players loosen further (tilting). TAG players tighten. Creates dynamic gameplay.
## Analysis Engine Overhaul (`analysis.ts`)
For each round where hero acted:
1. Reconstruct game state at that point
2. Call `evaluateDecision()` to get optimal play
3. EV loss = `EV(optimal action) - EV(hero's action)` from `evByAction`
4. Use `reasoning` for detailed mistake descriptions
5. Include bet sizing analysis when hero raised/bet
No random noise. Same hand → same analysis, every time.
